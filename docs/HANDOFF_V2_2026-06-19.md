## 2. Current Official Rule Set

The active profile is now:

- `strict_v2`

Defined in:

- [filters.yaml](/Users/teihiroshisakai/Documents/Codex/2026-06-17/pdbbind-index-users-teihiroshisakai-documents-codex/pdbbind_pl_pipeline/config/filters.yaml)

### 2.1 Complex definition

- Final output still keeps exactly:
  - one protein chain
  - one ligand item
- Multi-chain proteins are no longer deleted immediately.
- Instead, the pipeline computes ligand-protein heavy-atom contacts and picks a `primary_contact_chain`.
- Contact cutoff:
  - `5.0 A`
- If the primary chain does not dominate enough, the item is rejected as interface-dominated.
- Minimum primary contact fraction:
  - `0.70`

### 2.2 Hard exclusions

- `covalent complex`
- `incomplete ligand`
- `isomer`
- peptide-like `(<n>-mer)`
- unparseable ligand
- multi-component ligand
- no primary contact chain
- interface-dominated binding
- ligand molecular weight `<= 150`
- primary contact chain length `> 300 aa`

### 2.3 Sugar / nonsugar

Final output is binary only:

- `sugar`
- `nonsugar`

No sugar whitelist is used anymore.

The V2 sugar logic is structure-only and requires:

- a `5`- or `6`-membered ring
- at least `1` ring oxygen
- at least `2` exocyclic oxygens
- oxygen/carbon ratio at least `0.5`
- non-sugar heavy-atom fraction not dominating beyond `0.5`

Sugar rules live in:

- [sugar_rules.yaml](/Users/teihiroshisakai/Documents/Codex/2026-06-17/pdbbind-index-users-teihiroshisakai-documents-codex/pdbbind_pl_pipeline/config/sugar_rules.yaml)

### 2.4 Flexibility

V2 flexibility no longer uses only absolute rotatable bonds.

It uses:

- `ligand_rotatable_bonds`
- `ligand_rotatable_bonds_ratio = rotatable_bonds / heavy_atom_count`

Current thresholds:

- `rigid` if:
  - `rotatable_bonds <= 5`
  - and `rotatable_bonds_ratio <= 0.18`
- `flexible` if:
  - `rotatable_bonds >= 8`
  - or `rotatable_bonds_ratio >= 0.30`
- otherwise:
  - `intermediate`

Current profile policy:

- `intermediate` is excluded from the final main dataset.

### 2.5 Ligand and protein size constraints

- ligand molecular weight:
  - `> 150 Da`
- primary contact chain length:
  - `<= 300 aa`

## 5. Known Issues / Important Notes

### 5.1 Parallel rerun contamination happened before

This is the single most important procedural note.

At least once, rerunning `02/04/05` in parallel while `03` had not fully refreshed caused stale and fresh intermediate products to mix.
That produced false outputs such as:

- `3dx1` appearing in subset50 despite failing `primary_contact_chain_length <= 300`

If the next session reruns the pipeline, it should do so in this exact sequence:

1. `03_audit_structures.py`
2. `02_classify_ligands.py`
3. `04_score_nonsugar_flexibility.py`
4. `05_reduce_similarity.py`
5. `06_export_clean_complexes.py`
6. `07_validate_and_report.py`
7. `08_cluster_proteins_for_subset.py`
8. `09_select_subset50.py`
9. `10_select_subset100_nonsugar.py`

Do not restart only later phases unless it is clear the exact upstream files are already current.

### 5.2 Some RDKit warnings still appear

These include:

- explicit valence warnings on a few problematic ligands
- `MorganGenerator` deprecation warnings
- some `2D but z != 0` warnings when building subset fingerprints

These warnings do not currently stop the pipeline from finishing, but the codebase could be cleaned up further by:

- migrating to newer Morgan fingerprint APIs
- suppressing or handling more RDKit warnings
- explicitly tracking problematic ligands

### 5.3 subset100 still needs better small-ligand control

Current logic:

- `MW > 150`

Known remaining problem:

- halogen-heavy visually small ligands can still pass

Example:

- `1y4z`

Recommended next rule improvement:

- add `non_halogen_heavy_atom_count >= 10`
